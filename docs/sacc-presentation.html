But for Amiga, there's a problem…

AI agents need a tight feedback loop.
The Amiga doesn't give them one. Not by default.

The Challenge

AI agents are powerful coders — but they need to see what's happening.

Modern Platforms (easy for AI)

  • Write code → run → see output instantly
  • Error messages come right back
  • Tests run in milliseconds
  • Everything on the same machine

Amiga Development (hard for AI)

  • Cross-compile on macOS via Docker
  • Deploy binary to a separate machine (emulator)
  • No way to see the Amiga screen
  • Crashes take down the entire OS
  • No network stack, no SSH, no shell access
  • No memory protection — buffer overflows are fatal

The AI is Blind and Deaf

Without tooling, the AI can write Amiga code
but has zero ability to verify it works.

👁 ❌

Can't see the screen
Can't read crash info
Can't inspect memory
Can't press keys to test
Can't browse the filesystem

MCP: Giving AI Tools

Model Context Protocol — an open standard that lets AI call functions in the real world.

// During a conversation, Claude can call: amiga_build("defender") // cross-compile via Docker amiga_deploy("defender") // copy binary to Amiga amiga_launch("DH2:Dev/defender") // start the program amiga_screenshot() // SEE the Amiga screen amiga_input_key("space") // press a key amiga_call_hook("status") // ask the app for state amiga_get_var("score") // read a live variable amiga_set_var("lives", "99") // write a live variable amiga_inspect_memory("$DFF000") // read custom chips amiga_log("defender") // read debug logs

Every tool returns structured data that Claude can reason about and act on.

93 Tools Across 7 Categories

Build & Deploy — compile, deploy, launch, stop, run tests

Inspect — memory, tasks, libraries, screens, chip registers

Debug — breakpoints, step into/over, backtrace, registers

Graphics — screenshots, palette, copper list, sprites

Files — read, write, transfer, directory listing, assigns

App Integration — logs, variables, hooks, SnoopDos

Same tools available to both Claude (AI) and the Web UI (human).

How It All Connects

The Bridge Client Library

Your Amiga app links libbridge.a — giving it superpowers the AI can use.

// Initialize connection to bridge daemon ab_init("my_game"); // Remote logging — streams to host in real-time AB_I("Level %ld started", (long)level); AB_W("Low memory: %ld bytes", (long)free); AB_E("Crash in draw_enemies!"); // Register variables — host can read AND write these live ab_register_var("score", AB_TYPE_I32, &score); ab_register_var("lives", AB_TYPE_I32, &lives); ab_register_var("level", AB_TYPE_I32, &level); // Register hooks — functions the host can CALL remotely ab_register_hook("reset", "Reset game", hook_reset); ab_register_hook("next_level","Skip level", hook_next); ab_register_hook("add_life", "Extra life", hook_life); ab_register_hook("status", "Game status", hook_status); // Call every frame to process incoming commands ab_poll();

Why This Matters for AI

  • Logs — Claude reads your app's debug output to diagnose issues
  • Variables — Claude reads score/lives/state to verify game logic; writes them to set up test scenarios
  • Hooks — Claude calls "reset", "next_level", "kill_all" to test without playing

Also Available

  • Memory regions — expose buffers for host inspection
  • Performance profiling — frame timing, section timing
  • Test harness — AB_ASSERT macros with pass/fail reporting
  • Resource tracking — detect memory leaks

All over a serial link — works on real A500/A1200 hardware with a null-modem cable.

The Agentic Development Loop

1
Prompt: "Build me a Planet Patrol game with parallax…"
2
Clarify: Claude asks: scanner? enemy types? resolution?
3
Plan: Explores codebase, studies existing patterns
4
Write: Creates game.h, main.c, draw.c, input.c, sound.c…
5
amiga_build → compile errors? Fix and retry
6
amiga_deploy + amiga_launch → run on Amiga
7
amiga_screenshot → Claude sees the game!
8
amiga_input_key + amiga_call_hook("status")
9
User: "Too slow!" → Claude profiles, optimizes, rebuilds
10
User: "Add smart bombs" → Claude adds feature, tests it

MCP Tools Close the Loop

amiga_build — compiler errors return to Claude

amiga_screenshot — Claude sees the actual display

amiga_call_hook — "status" returns game state

amiga_get_var — read score, lives, frame count

amiga_set_var — set lives=0 to test game over

amiga_input_key — automated play-testing

amiga_log — debug output in real-time

What Makes It Work

The key insight: every step where the AI needs feedback has an MCP tool that provides it. The loop is write → build → deploy → observe → adjust, and the AI can do it all autonomously.
